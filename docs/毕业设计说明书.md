# 电商用户评论情感倾向分析平台

## 毕业设计说明书

---

**题目名称：** 电商用户评论的情感倾向分析  
**开发周期：** 第 1 周至第 18 周  
**技术栈：** Python、FastAPI、SnowNLP、jieba、SQLite、HTML/CSS/JavaScript、Chart.js

---

## 摘要

电商平台每天产生大量用户评论，这些评论里既有对商品质量的肯定，也有对物流、售后等方面的抱怨。如果完全依靠人工逐条阅读，不仅效率低，而且很难从整体上把握用户情绪变化。本课题设计并实现了一个面向电商场景的评论情感分析平台，用户可以将评论逐条输入、批量粘贴或通过 CSV 文件导入，系统自动给出情感得分和正/中/负分类，并在仪表盘上展示统计图表与关键词分布。

系统后端采用 FastAPI 框架，情感分析核心使用 SnowNLP 计算中文文本情感概率，分词与关键词统计使用 jieba 完成；数据持久化采用 SQLite 数据库。前端为单页 Web 应用，通过 RESTful 接口与后端交互，使用 Chart.js 绘制情感分布、商品对比和趋势折线图。测试结果表明，系统能够稳定完成评论导入、分析、筛选、导出等常见操作，适合作为电商运营辅助工具，也满足本科毕业设计对“完整系统 + 文档 + 功能截图”的要求。

**关键词：** 情感分析；电商评论；SnowNLP；FastAPI；数据可视化

## 第一章 绪论

### 1.1 研究背景

近年来网络购物已成为居民日常消费的重要方式。商品详情页下方的用户评价，往往直接影响潜在买家的购买决策，也是商家改进产品和服务的重要依据。一条评论通常只有几十到几百字，但当一个店铺或平台积累成千上万条评论后，人工归纳就变得十分困难。

情感分析（Sentiment Analysis）是自然语言处理中的经典任务，目标是从文本中判断作者的态度倾向。将这一技术应用到电商评论场景，可以把杂乱的文字信息转化为可统计、可对比的数据，例如某款商品近期负面评论是否增多、用户主要在抱怨“续航”还是“做工”等。对于本科毕业设计而言，该题目既有明确的应用背景，又不需要昂贵的硬件设备，开发周期可控。

### 1.2 研究意义

本系统的意义主要体现在三个方面：

1. **业务层面**：帮助运营人员快速了解用户满意度，为客服跟进和商品优化提供参考。
2. **技术层面**：综合运用 Web 开发、数据库、中文 NLP 和可视化技术，体现完整的软件工程流程。
3. **教学层面**：题目边界清晰，便于划分需求、设计模块、编写测试和撰写论文。

### 1.3 国内外研究现状

国外较早将情感分析用于商品评论，Amazon、Yelp 等平台均公开过相关数据集。常见方法包括基于词典的规则方法、传统机器学习方法（如 SVM、朴素贝叶斯）以及深度学习方法（如 LSTM、BERT）。国内电商评论以中文为主，需要额外处理分词、否定词和口语表达等问题，SnowNLP、百度 NLP 开放平台、HanLP 等工具在中文情感分析中应用较多。

本设计采用 SnowNLP 作为核心分析引擎，主要考虑到其开源、轻量、易于本地部署，适合毕业设计演示环境。若后续需要更高精度，可在保留系统架构的前提下替换为微调后的深度学习模型。

### 1.4 研究内容与论文结构

本文主要工作包括：需求分析、系统总体设计、数据库与接口设计、情感分析模块实现、前端页面开发、系统测试以及总结展望。论文共分六章：第一章绪论；第二章相关技术；第三章需求分析；第四章系统设计；第五章系统实现与测试；第六章总结与展望。

## 第二章 相关技术介绍

### 2.1 FastAPI 框架

FastAPI 是基于 Python 3 的现代 Web 框架，底层基于 Starlette 和 Pydantic。它具有自动生成接口文档、类型校验严格、异步支持良好等特点。本系统使用 FastAPI 提供 REST API，并托管前端静态资源，部署时只需启动一个 Python 进程即可。

### 2.2 SnowNLP 情感分析

SnowNLP 是一个专门针对中文文本处理的 Python 库，其中 `sentiments` 方法返回 0 到 1 之间的情感概率值，越接近 1 表示越偏向正面情绪。该方法基于朴素贝叶斯模型，在通用中文语料上训练，对电商评论这类短文本具有较好的可用性。

本系统将得分划分为三档：

| 得分区间 | 情感标签 |
|---------|---------|
| ≥ 0.6   | 正面    |
| 0.4 ~ 0.6 | 中性  |
| ≤ 0.4   | 负面    |

阈值可根据实际数据分布调整。

### 2.3 jieba 分词与关键词提取

关键词模块使用 jieba 对评论进行精确模式分词，过滤长度小于 2 的字符、数字符号以及常见停用词后，统计词频并返回 Top N。该方案实现简单，能够直观反映用户关注点，例如“续航”“客服”“包装”等。

### 2.4 前端与可视化

前端未引入复杂构建工具，采用原生 HTML/CSS/JavaScript，降低环境配置成本。图表部分使用 Chart.js，支持饼图、柱状图和折线图，用于展示情感占比、不同商品平均得分以及按日期聚合的情感趋势。

### 2.5 SQLite 数据库

SQLite 是一种嵌入式关系型数据库，无需独立安装数据库服务，适合本地演示和小规模数据存储。系统通过 SQLAlchemy ORM 定义 `reviews` 表，字段包括商品名称、评论内容、情感得分、情感标签、数据来源和创建时间等。

## 第三章 系统需求分析

### 3.1 可行性分析

**技术可行性：** 所选技术均为成熟开源方案，文档齐全，社区案例丰富。  
**经济可行性：** 全部软件可免费使用，运行环境为一台普通 PC 即可。  
**操作可行性：** 界面采用侧边栏导航，功能分区明确，非技术人员经过简单说明即可上手。

### 3.2 功能需求

系统应实现以下功能：

| 编号 | 功能名称       | 功能描述 |
|-----|--------------|---------|
| F1  | 单条评论分析   | 输入一条评论，即时返回情感得分、倾向和关键词 |
| F2  | 分析结果入库   | 将单条分析结果保存到数据库 |
| F3  | 批量文本导入   | 支持多行粘贴，格式为“商品\|评论”或纯评论 |
| F4  | CSV 文件导入   | 上传含“评论内容”列的 CSV 文件并批量分析 |
| F5  | 数据概览       | 展示评论总量、正/中/负数量及平均得分 |
| F6  | 图表统计       | 情感分布饼图、商品得分柱状图、趋势折线图 |
| F7  | 评论列表管理   | 按情感、商品、关键词筛选，支持删除 |
| F8  | 关键词洞察     | 按全部或单一情感类别统计高频词 |
| F9  | 数据导出       | 将分析结果导出为 CSV 文件 |
| F10 | 示例数据加载   | 一键导入内置电商评论样例，便于答辩演示 |

### 3.3 非功能需求

1. **响应时间：** 单条评论分析应在 1 秒内完成。  
2. **可维护性：** 前后端分离为 API + 静态页面，模块职责清晰。  
3. **可扩展性：** 情感引擎可替换，数据库可迁移至 MySQL/PostgreSQL。  
4. **易用性：** 页面布局简洁，关键操作配有提示文字。

### 3.4 用例分析

主要参与者为“运营分析人员”。典型用例包括：导入评论、查看情感分布、筛选负面评论、导出报告。系统用例图可在论文排版阶段使用 Visio 或 draw.io 绘制，此处从略。

## 第四章 系统设计

### 4.1 系统总体架构

系统采用 B/S 三层结构：

```
┌─────────────┐      HTTP/JSON      ┌─────────────┐      ORM       ┌─────────────┐
│  浏览器前端  │  ◄──────────────►  │  FastAPI    │  ◄──────────►  │   SQLite    │
│  HTML/JS    │                     │  业务接口    │                │  reviews 表  │
└─────────────┘                     └──────┬──────┘                └─────────────┘
                                           │
                                    ┌──────▼──────┐
                                    │  分析模块    │
                                    │ SnowNLP+jieba│
                                    └─────────────┘
```

**表现层：** 负责页面展示与用户交互。  
**业务层：** 处理评论增删查、统计分析、文件解析等逻辑。  
**数据层：** 持久化评论及分析结果。  
**算法层：** 封装情感打分与关键词提取，供业务层调用。

### 4.2 功能模块划分

1. **评论采集模块：** 单条录入、批量粘贴、CSV 上传。  
2. **情感分析模块：** 文本清洗、SnowNLP 打分、情感标签映射。  
3. **数据管理模块：** 列表查询、条件筛选、删除、清空、示例加载。  
4. **统计分析模块：** 汇总计数、按商品聚合、按日期趋势统计。  
5. **可视化模块：** 图表渲染与关键词条形展示。  
6. **导出模块：** 生成 UTF-8 CSV 下载。

### 4.3 数据库设计

**表名：reviews**

| 字段名        | 类型         | 说明 |
|--------------|-------------|------|
| id           | INTEGER PK  | 主键，自增 |
| product_name | VARCHAR(128)| 商品名称 |
| content      | TEXT        | 评论正文 |
| score        | FLOAT       | 情感得分 0~1 |
| sentiment    | VARCHAR(16) | positive/neutral/negative |
| source       | VARCHAR(32) | 来源：manual/batch/upload/sample |
| created_at   | DATETIME    | 创建时间 |

### 4.4 接口设计（部分）

| 方法 | 路径 | 说明 |
|-----|------|------|
| POST | /api/analyze | 单条分析，不入库 |
| POST | /api/reviews | 新增评论并分析 |
| POST | /api/reviews/batch | 批量新增 |
| POST | /api/reviews/upload | CSV 上传 |
| GET  | /api/reviews | 条件查询列表 |
| GET  | /api/stats | 统计数据 |
| GET  | /api/keywords | 关键词列表 |
| GET  | /api/export | 导出 CSV |
| DELETE | /api/reviews/{id} | 删除单条 |

完整接口可通过启动服务后访问 `http://localhost:8000/docs` 查看 Swagger 文档。

### 5.2 核心代码说明

**情感分析函数（backend/sentiment.py）：**

系统先对文本做空白字符规整，再调用 `SnowNLP(cleaned).sentiments` 获取概率值，最后根据预设阈值映射为 positive、neutral、negative 三类。该函数被单条分析、批量导入、文件上传等多个接口复用，避免逻辑重复。

**统计接口（backend/main.py `/api/stats`）：**

从数据库读取全部评论，计算正/中/负数量与平均得分；使用 SQL `GROUP BY` 按商品统计评论数和平均得分；按 `created_at` 日期字段聚合得到趋势数据，供前端折线图使用。

**前端导航（frontend/js/app.js）：**

通过 `switchSection` 函数切换五个功能页面，并在进入仪表盘、评论列表、关键词页时自动拉取最新数据，保证界面与后端状态一致。

## 第六章 总结与展望

### 6.1 工作总结

本课题完成了电商用户评论情感倾向分析平台的设计与实现。系统实现了评论采集、情感分析、可视化统计、关键词提取和数据导出等功能，代码结构清晰，能够支撑毕业答辩演示。通过本次开发，加深了对中文文本处理流程、Web 后端设计和前后端协作方式的理解。

### 6.2 不足与改进方向

1. **分析精度有限：** SnowNLP 为通用模型，对电商领域专有表达（如“踩雷”“真香”）识别不足，可引入领域词典或微调 BERT 模型。  
2. **缺少用户权限：** 当前为单用户演示版，未实现登录与角色管理。  
3. **趋势数据依赖入库时间：** 示例数据创建时间相近，真实环境应按评论原始时间字段统计。  
4. **未做性能压测：** 大批量导入时尚未验证并发与内存占用。

### 6.3 展望

后续可对接真实电商平台 API，定时拉取增量评论；增加主题聚类（LDA）和方面级情感分析，例如分别判断“物流”“质量”“价格”的情感；并部署到云服务器，供多人协同使用。
